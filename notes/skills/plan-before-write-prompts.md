# 重活先 plan、轻活直接做的提示词设计

## 问题

LLM 拿到"再写一章"这种生成型请求,默认行为是**直接开始 60-120 秒的长生成**。
对用户来说:
- 几十秒"思考中…"才开始出字 → 怀疑卡死(见 `gateway-buffering-heartbeat.md`)
- 真出来了不一定是想要的:方向、字数、视角、关键事件不对
- 改起来等于重生成,代价线性叠加

而创作类工具的核心 UX 是 **"和 AI 一起决定写什么、再写"**,不是"AI 单方面交付"。

## 套路

System prompt 里硬性插一段 plan 协议,把"先讨论、再动笔"变成默认行为:

```
## 对话原则(非常重要)

**重活前先 plan**:当用户说"写一章 / 写大纲 / 接着写 / 改一段"等
明显需要生成大段内容的请求时,**禁止直接动手写正文**。先用 1-2 段简短回复
做这几件事:
  a) 复述前文(从大纲、上一章结尾、稳定设定)给 1-2 句具体建议雏型;
  b) 给 2-3 个发展方向 / 风格 / 视角 / 关键事件的选择,让用户挑;
  c) 询问还需要确认的关键信息(目标字数、是否引入新角色等),最多 1-2 条最关键的。
  然后停下来等用户回复,**这一轮不要调用任何写章节/写大纲的工具**。

**确认后再动手**:用户回复选择了第几条 / 直接开始 / 搞定了之后,**写之前**先用一句话总结方向("好的,我按 X 思路来写第 N 章,目标 ~M 字,重点在 …"),再调用写入工具。

**例外:续写 skill / 明确指令直接写**
  - 系统使用「续写」专用 prompt(system_continue)生效时,可直接动手
  - 用户明确说"不用问了直接写" / "你定" → 跳过 plan,直接做
  - 微改、润色、单段重写、问问题、查信息 → 不用 plan

**永远先发声**:即便 plan 出来要等用户,也要先返回一段文字("我先理一下…"),
不要直接 silent + 工具调用。第一道 token 就是用户的"AI 活着"信号。
```

### 关键点

1. **正面+负面双向写**:既说"做什么"(plan、确认、复述),也说"不做什么"(不调工具)。
   光写正面,LLM 仍可能"我先解释一句然后顺便写完",防御性写法是必须的。

2. **明确豁免**:续写、单段润色、回答问题这些**不该被 plan 拖慢**的场景必须列出。
   否则 LLM 会过度保守,所有请求都先问一遍,反而烦人。

3. **"永远先发声"**:这是 TTFT UX 的硬约束。
   即使要调工具,也先用 1-2 句承接性文字开个头("我看了大纲,目前来到第 5 章……"),
   LLM 流首字到达 = 用户从"等待"切到"AI 在干活"。

4. **plan 里给 2-3 个选项**(不是 5+):多了用户不愿读、要二次筛;
   少于 2 不算 plan。**带上序号**让用户能"3,2500 字"一句话回。

## 反例

- ❌ "先问清楚再做"这种空话:LLM 不知道"清楚"的标准,问 5 轮还在确认。
  必须给具体动作:"复述 + 给 2-3 选项 + 1-2 关键问题"。
- ❌ 把 plan 写在 user message 里、靠用户每次手动触发:用户写小说时不会记得 prompt 工程。
  写在 system prompt 让它常驻。
- ❌ 不写豁免:"反提取人物" / "把这段润色一下" 都要先 plan,体验灾难。
  豁免列表是 plan-mode 能用的前提。
- ❌ 让 plan 和写作工具用同一个 system prompt:专用「续写」流程被强制 plan,
  和"用户已经选好,后端代理直接续写"的语义冲突。
  → 起码区分 `system_creator`(plan-mode)和 `system_continue`(直接动手)。

## 验证标准

实测对话应当呈现:

```
user> 再写一章
AI>   好的,基于你前面的剧情,我给你 3 个发展方向:
       1) 主角进城接受任务,引入新人物
       2) 回村寻找线索,与师傅冲突
       3) 山中遇险,触发觉醒
       你倾向哪个?目标字数大约多少?
       (这一轮没有任何 tool_call)

user> 3,就 2500 字吧
AI>   好的,我按"山中遇险触发觉醒"的方向来写第 N 章,
       目标 ~2500 字。重点放在 …
       [tool_call: add_chapter ...]
```

第一轮 0 tool_call、第二轮带 tool_call,且写之前有总结句 = 通过。
任何一条不满足,system prompt 就要再加防御。

## 真实位置

- Plan 协议: `backend/app/prompts/templates.py` 的 `_PLAN_MODE` 块
- 注入: `system_creator(ctx)` 函数
- 续写专用(豁免 plan): `system_continue(ctx)`
